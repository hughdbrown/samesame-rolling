  Why the merging doesn't extend

  The rolling hash extension works by sliding the window one line at a time and
  checking if consecutive windows also match. For the window at position 1134
  (0-indexed, line 1135), a 15-line window captures both the model and sessionID
   blocks. Since XOR is order-independent, the hash matches despite the swap.

  But when the window slides to position 1135 (line 1136):
  - Region 1 starts with if model.Valid {
  - Region 2 starts with if sessionID.Valid {

  The composite BlockGroupKey includes first_hash, so these windows have
  different keys and don't match. The same happens at positions 1136-1139 ‚ÄĒ the
  swap means corresponding lines differ.

  At position 1140 (line 1141, if provider.Valid {), the content IS genuinely
  identical from there on. Those windows match and form a separate run. With
  --match=15, this produces a second group of 16 lines (15 + 2 - 1, from windows
   at positions 1140-1141).

  So the result is two separate runs that can't be merged because positions
  1135-1139 have a gap:
  1. Position 1134 alone ‚Üí 15-line group (false positive from XOR permutation)
  2. Positions 1140-1141 ‚Üí 16-line group (genuine duplicate from provider
  onward)

  With --match=22, there's only one window position (1134), the XOR of all 22
  lines matches (same multiset), and first/last hashes match (both }), so it
  reports 22 lines.

  In summary: The tool is actually working correctly here. The "22-line match"
  is a false positive ‚ÄĒ the code isn't actually identical, just permuted. The
  15-line and 16-line groups more accurately reflect what's happening (one false
   positive from XOR, one genuine duplicate of the tail).
